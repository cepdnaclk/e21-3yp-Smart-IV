System Overview & Architecture

The SmartIV cloud integration creates a robust, real-time, serverless medical monitoring system. The pipeline enables high-frequency bedside telemetry ingestion, automated cloud-level alert routing, push notifications, and a responsive nursing dashboard on both desktop and mobile apps.

[ Bedside Units (ESP32) ] │ (BLE / RF) [ Receiver Dongle (ESP32) ] │ (USB Serial Ingestion) [ Tauri Desktop App (Rust Backend) ] │ (Mutual TLS (mTLS) over MQTT) [ AWS IoT Core ] │ (IoT Rules Engine SQL Routing) ┌────┴────────────────────────┐ ▼ ▼ [ DynamoDB Telemetry Table ] [ AWS Lambda Alert Engine ] ▲ │ (SNS API) │ (API Gateway REST) ▼ │ [ AWS SNS / Firebase Cloud Messaging (FCM) ] │ │ (Push Notification) [ React Native Mobile App ] ◄──────┘ │ (AWS Amplify & Cognito Auth) ▼ [ Real-time Nurse Dashboard ]
Phase 1

AWS IoT Core & mTLS Connectivity

To connect the local Tauri desktop monitoring station to the cloud, we migrated the desktop app from a mock local broker to AWS IoT Core using mutual TLS (mTLS) for authorization. The desktop app must authenticate itself using a unique cryptographic identity consisting of three files: Amazon Root CA 1, Device Certificate, and a Private Key.

⚙️ Implementation Steps

  1. IoT Thing Setup: Created a single IoT Thing named smartiv-desktop-station-1 in AWS IoT Core (ap-south-1 Mumbai region).
  2. Security Credentials: Generated and downloaded the mutual TLS keys.
  3. IoT Policy: Attached a restrictive IoT Policy named SmartIVDesktopPolicy allowing secure iot:Connect, iot:Publish, and iot:Subscribe actions under the smartiv/* resource namespace.
  4. Rust Backend Connection: Configured mqtt.rs in the Tauri app to load certificate bytes using std::fs::read and pass them into rumqttc AsyncClient.

🚨 Hurdles & Resolutions

Hurdle 1.1: Missing Rust Compiler Environment
Cause

The desktop development machine did not have the Rust compiler (rustc / cargo) installed or added to the path, blocking Tauri builds.

Resolution

Downloaded and initialized the rustup installer, configured the stable toolchain, and refreshed environment path variables by restarting the shells.

Hurdle 1.2: Missing MSVC C++ Linker link.exe
Cause

Rust builds on Windows require the MSVC C++ Linker tool to pack system DLLs and execute compilation.

Resolution

Orchestrated the installation of Visual Studio Build Tools with the C++ desktop workload, forcing compiler registration and setting the toolchain default:

rustup default stable-x86_64-pc-windows-msvc
Hurdle 1.3: Certificate Path Mismatch & Key Exposure
Cause

Relative paths like certs/ cert solved incorrectly during tauri dev modes because the binary working directory runs inside src-tauri/. Also risk of committing keys to Git.

Resolution

Decoupled the binary from compile-time embedding. Moved the certificate folder under src-tauri/certs/, modified Rust code to dynamically read bytes at runtime, and updated .gitignore immediately to block certificate file tracking.

Phase 2

Data Routing: Rules Engine & DynamoDB

A telemetry packet arriving at AWS IoT Core must be processed and persisted with zero latency. Running an active virtual machine (EC2) to read and store these packets is expensive and prone to downtime. Instead, we utilized the serverless **IoT Rules Engine** and **Amazon DynamoDB** to automatically intercept MQTT messages and route them directly into the database.

⚙️ Implementation Steps

  1. Database Tables: Created smartiv-telemetry and smartiv-alerts tables using a partition key of bedId (String) and a sort key of ts (String).
  2. IoT Rules: Created SmartIVStoreTelemetry to extract streams: 
    SELECT *, timestamp() AS receivedAt FROM 'smartiv/+/+/telemetry'
  3. IAM Roles: Formulated SmartIVIoTRulesRole giving IoT Core permission to perform dynamodb:PutItem on the database.

🚨 Hurdles & Resolutions

Hurdle 2.1: Schema Mismatch Silently Dropping Packets
Cause

The IoT Rule SQL expected exact property mappings like bedId and ts, but Rust models serialized with varying camelCase shapes, while the simulator published mismatch properties.

Resolution

Decorated the Rust structures in Tauri with #[serde(rename_all = "camelCase")] and validated matching attributes in TypeScript to ensure perfect serializability.

Hurdle 2.2: DynamoDB Composite Key Sorting Performance
Cause

Querying historical telemetry for a single bed took too long or required expensive full-table scans.

Resolution

Configured bedId as Partition Key and ts (ISO 8601 timestamp string) as Sort Key. Because ISO strings sort alphabetically, querying history chronologically uses simple fast query operations instead of full-table scans.

Phase 3 & 4

AWS Lambda & Push Alerts (SNS/FCM)

To run real-time background processing on bedside telemetry, we connected a Python 3.12 Lambda function (SmartIVAlertHandler) to the IoT Rules engine trigger. If the incoming payload reports a blockage or empty bag, the Lambda writes it to the alerts database and pushes an urgent notification to nurse smartphones via Amazon SNS and Firebase Cloud Messaging (FCM).

🚨 Hurdles & Resolutions

Hurdle 3.1: The Duplicate Alert Notification Storm
Cause

Because ESP32 bedside units stream telemetry continuously (every 5 seconds), a sustained blockage would trigger duplicate push notifications to the nurses every 5 seconds, causing severe notification fatigue.

Resolution

Implemented a database-driven deduplication lookup inside the Lambda function using the boto3 SDK before triggering notifications:

existing = table.query(
    KeyConditionExpression='bedId = :bid',
    FilterExpression='alertType = :at AND attribute_not_exists(resolvedAt)',
    ExpressionAttributeValues={':bid': bed_id, ':at': alert_type},
    Limit=1
)
if existing['Items']:
    print("[DEDUP] Alert already active. Skipping.")
    return {'statusCode': 200, 'body': 'Duplicate alert suppressed'}
Phase 5

API Gateway REST Endpoints

We built a serverless public REST API using Amazon API Gateway (HTTP API) to expose endpoints to the mobile app. The endpoints proxy calls to specific Lambda functions retrieving DynamoDB history or acknowledging active alerts.

Route Method Lambda Function Purpose
/beds/{bedId}/history GET SmartIVGetTelemetryHistory Fetches historical charts
/alerts/acknowledge POST SmartIVAcknowledgeAlert Nurse resolves active alerts
/notifications/register POST SmartIVRegisterPushToken Registers mobile push tokens

🚨 Hurdles & Resolutions

Hurdle 5.1: 502 Bad Gateway Proxy Errors
Cause

API Gateway expects integration proxy responses to be structured with strict JSON keys containing statusCode and headers, but Lambdas initially returned flat JSON objects.

Resolution

Structured all Lambda return dictionaries to strictly comply with API Gateway Proxy requirements, returning status codes and explicit headers.

Hurdle 5.2: CORS Permission Blocks on Mobile Clients
Cause

HTTP requests to the API from React Native clients were rejected by the browser or mobile sandbox due to missing Cross-Origin Resource Sharing (CORS) headers.

Resolution

Enabled CORS in the API Gateway dashboard and explicitly injected CORS header dictionaries inside the Lambda response payloads:

'headers': {
    'Content-Type': 'application/json',
    'Access-Control-Allow-Origin': '*',
    'Access-Control-Allow-Methods': 'GET,POST,OPTIONS'
}
Hurdle 5.3: Composite Keys Preventing Simple Alert Resolves
Cause

The alerts table in DynamoDB requires both Partition Key (bedId) and Sort Key (ts). The mobile app initially tried to resolve alerts using a flat numeric ID, causing database validation crashes.

Resolution

Re-designed the backend `/alerts/acknowledge` endpoint and updated `apiService.ts` to pass the unique compound composite keys (both bedId and ts). This allowed the Lambda to perform explicit, validated database updates:

table.update_item(
    Key={'bedId': bed_id, 'ts': ts},
    UpdateExpression='SET resolvedAt = :r, resolvedBy = :n',
    ExpressionAttributeValues={':r': datetime.now().isoformat(), ':n': resolved_by}
)
Phase 6

AWS Cognito: Nurse Authentication

To secure patient telemetry and audit alert responses, we integrated AWS Cognito. We configured a User Pool to handle nurse registration/attributes and an Identity Pool to federate those logins into temporary IAM credentials. These temporary credentials grant the mobile app secure access to the AWS IoT broker over WebSockets.

⚙️ Implementation Steps

  1. User Pool Setup: Configured a User Pool named SmartIVUserPool with custom:ward and custom:role attributes.
  2. Identity Pool Setup: Created SmartIVIdentityPool to exchange Cognito User Pool tokens for temporary AWS temporary keys.
  3. Amplify Configuration: Integrated the AWS Amplify client block inside `app/_layout.tsx` to handle authentication states automatically.

🚨 Hurdles & Resolutions

Hurdle 6.1: Cognito Client Secret Incompatibility on Mobile
Cause

Initial Cognito App Client generation was configured with "Generate client secret" checked, causing Amplify connections on mobile to crash immediately due to missing secure secret stores.

Resolution

Reconfigured the Cognito App Client, ensuring that **"Generate client secret"** was **disabled** during creation since React Native compiled binaries cannot securely store application client secrets.

Hurdle 6.2: Enforced "Force Change Password" Challenge Hang
Cause

Users created via the AWS Console are assigned a status of FORCE_CHANGE_PASSWORD, which caused mobile sign-ins to hang on a loading screen because the interface had no panel to accept a new password.

Resolution

Handled this transitional state programmatically inside authService.ts. If the next auth step requires a new password, the service programmatically confirms the transition by re-submitting the temporary password as the new permanent password to bypass this friction during development:

} else if (nextStep && nextStep.signInStep === 'CONFIRM_SIGN_IN_WITH_NEW_PASSWORD_REQUIRED') {
    const confirmResult = await confirmSignIn({ challengeResponse: password });
    if (confirmResult.isSignedIn) {
        await this.checkSession();
    }
}
Phase 7

Mobile MQTT via WebSockets (SigV4)

To make the mobile app truly responsive and replace high-overhead REST polling, we implemented a direct MQTT over WebSocket connection to AWS IoT Core using the nurse's Cognito credentials via the @aws-amplify/pubsub plugin. WebSocket URLs are mathematically signed with temporary credentials, providing secure, real-time telemetry streaming.

🚨 Hurdles & Resolutions

Hurdle 7.1: Node.js Native Module Network Crashes
Cause

Standard npm mqtt libraries rely on Node.js native net and tls socket modules, causing instant crashes inside mobile React Native/Expo Go runtimes.

Resolution

Switched to `@aws-amplify/pubsub` using WebSockets, which compiles smoothly on mobile runtimes and establishes signed web connections to the IoT Broker endpoint.

Hurdle 7.2: Cognito IoT "Not Authorized" Connect Failures
Cause

The Cognito Identity Pool's authenticated IAM role had no explicit rights to connect, subscribe, or publish to AWS IoT Core, throwing silent auth rejections.

Resolution

Attached an inline IAM policy named SmartIVMobileIoTPolicy directly to the `Cognito_SmartIVIdentityPoolAuth_Role` role, granting connect and subscribe rights for the smartiv/* topic path.

Hurdle 7.3: The Reconnect Loop & Paho AMQJS0011E Crash
Cause

The React Native useMqtt.ts hook listened to the entire nurse object in its dependency array. Since objects are recreated on every state shift, the effect triggered continuously, initiating a rapid connect-disconnect storm. Paho MQTT crashed with AMQJS0011E Invalid state not connected, rendering a white screen.

Resolution

First, we optimized useMqtt.ts to listen only to the primitive ward string rather than the full object. Second, we added a concurrency lock in mqttService.ts to block double-handshakes:

if (subscription || isConnecting) return;
isConnecting = true;
Hurdle 7.4: Blue/White Loading Screen Routing Guard Hang
Cause

If an authenticated user opened the app, they landed on the root route `/` (the loading spinner `index.tsx`) which sits outside both standard groups. Lacking a redirection rule, they remained stuck on the loading page.

Resolution

Updated `_layout.tsx` routing guard to track `inAppGroup`. When an authenticated user is detected outside the app folder, they are programmatically pushed directly to the Ward Dashboard (`/(app)/ward`), clearing the hang.

Phase 8

Hardcoding Endpoints & Production Polish

To eliminate developer and operator friction during deployments, we integrated the production-ready AWS IoT Core endpoint a2qzkepylp0vfe-ats.iot.ap-south-1.amazonaws.com and Thing Name smartiv-desktop-station-1 as compile-time defaults in Tauri Rust. We also added an auto-refresh hook to the ward status dashboard to seamlessly synchronize DynamoDB state changes.
